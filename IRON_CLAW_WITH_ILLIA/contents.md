## 밋업의 핵심 질문

이 밋업은 하나의 관찰에서 출발했다: **AI 경쟁의 전선이 "모델 성능"에서 "실행 인프라(하네스 레이어)"로 이동하고 있다.** "Attention Is All You Need"의 공저자 Illia Polosukhin이 모델 연구를 떠나 Rust 기반 에이전트 하네스 IronClaw를 직접 만들고 있다는 사실이 그 상징이다.

호스트 정구봉은 세 가지 질문을 던졌다:
1. 왜 최정상급 모델 연구자가 하네스 레이어를 만들고 있는가?
2. 왜 보안과 Rust 기반 샌드박싱을 강조하는가?
3. 왜 하네스 레이어가 오픈소스로 수렴하고 있는가?

## 산업 지형의 변화

2026년 4월 기준, 하네스 레이어를 둘러싼 움직임이 동시다발적으로 일어나고 있다:

- **Anthropic** — Claude Managed Agent 출시. 기존 하네스 도구와 유사한 기능을 자사 제품에 내장.
- **OpenAI** — Agent SDK에 샌드박스 기능 추가. Claude Code나 오픈소스 하네스의 아키텍처를 미러링.
- **Nous Research** — 모델 파인튜닝 조직에서 갑자기 방향을 틀어 Hermes Agent를 출시. 2개월 만에 대규모 사용자 확보.
- **Cognition** — Windsurf(에디터)와 Devin(클라우드 에이전트)을 통해 엔터프라이즈 에이전트 시장을 정의 중.

이전에는 Claude Code 같은 프론티어 랩 제품이 페이스를 주도했지만, 이제는 **오픈소스 하네스가 선도하고 프론티어 랩이 따라가는** 역전 현상이 발생하고 있다.

## 코딩 워크플로우의 진화 단계

이 밋업의 발표들을 종합하면, 코딩 워크플로우는 다음과 같이 진화하고 있다:

### 1단계: 바이브 코딩 (2024~2025 초)
- 커서(Cursor), 코파일럿(Copilot) 등으로 코드 자동완성
- 사람이 프롬프트를 주고, AI가 코드를 생성하면, 사람이 읽고 커밋

### 2단계: 에이전트 코딩 (2025~2026 초)
- Claude Code, OpenAI Codex 등 터미널 기반 에이전트 등장
- 사람이 지시하면 에이전트가 파일을 직접 수정하고 테스트까지 실행
- 여전히 사람이 결과를 리뷰하고 PR을 올림

### 3단계: 자율 워크플로우 (2026~ 현재 진행 중)
- **Jeff Wang이 설명한 Devin/Windsurf의 목표 워크플로우:**
  1. GitHub에 이슈가 등록되면 에이전트가 자동으로 계획을 수립
  2. 개발자는 계획을 리뷰하고 방향만 조정 ("잠깐, 이쪽으로 가야 해")
  3. 에이전트가 코드 작성 → 유닛 테스트 실행 → 웹 브라우저로 기능 검증 → 녹화본을 사람에게 전송
  4. CI가 충분히 좋으면 머지까지 자동화 가능
- Jeff의 예측: **"1년 안에 아무도 코드 리뷰를 하지 않을 것"**
- IDE를 열지 않고도 이슈 → 코드 → 테스트 → PR까지 완결되는 루프

### 그 너머: 에이전트 커머스
- Illia가 제시한 비전: 에이전트끼리 협상하고, 에스크로로 자금을 관리하고, 작업을 수행한 뒤 결과를 전달하는 **에이전트 마켓플레이스**
- Trust Execution Environment(TEE)에서 실행하면 에이전트의 전체 실행 추적을 암호화 서명으로 검증 가능
- Fiverr 대비 5배 저렴하고, 1시간 만에 완료되는 실험 결과를 공유

## 현실의 교훈: Yeachan Heo의 "가재 운영기"

이론과 비전 사이에는 현실이 있다. Yeachan Heo는 4마리의 에이전트("가재")를 실제로 운영하면서 얻은 교훈을 공유했다:

### 멀티 에이전트의 함정
- 에이전트를 여러 개 붙여놓으면 서로 일을 떠넘기고, 오케스트레이션만 하다가 결국 아무것도 안 함
- R&R이 명확하지 않으면 멀티 에이전트는 오히려 비효율적
- 가재 100마리를 띄워놓고 스타트업을 만들었다는 사람은 "대표 사기이거나 굉장히 잘못 쓰고 있는 것"

### 모니터링의 중요성
- 크론잡으로 마케팅 테스트를 시켜놓고 가만히 있었더니, 5일째 프로세스가 죽어있는 줄도 몰랐음
- 와치잡(watchjob)을 안 붙여놓은 것이 원인
- 클라우드 환경의 모니터링/텔레메트리 인프라가 이런 사고를 방지할 수 있음

### 프롬프트 인젝션 사고
- 이슈에 "Alert"라고 빨갛게 써놓으면 절대 머지하면 안 되는 건데, 에이전트가 인젝션에 당해서 릴리즈가 돼버림
- 아침에 일어나니 이미 배포 완료 → 프롬프트 인젝션 디펜스의 중요성

### 안티프래질(Antifragile) 마인드셋
- 에이전트 시스템은 깔끔하게 동작할 거라고 기대하면 안 됨
- 오히려 LLM이 확률적으로 스티어링을 잘 해주기 때문에, 계속 고쳐가면서 쓰는 리질리언스가 핵심
- 신입 교육하듯 한 달은 옆에서 붙어서 심력을 써야 뭐가 되기 시작함

### 셀프 러닝 루프의 한계
- 에르메스(Hermes) 등의 셀프 러닝 루프가 혁신적으로 보이지만, 2개월 뒤면 불만이 폭증할 것으로 예측
- 지금은 메모리를 전부 한 컨텍스트에 넣어도 동작하지만, 곧 컨텍스트 오버플로우가 발생
- LLM에는 컨텍스트를 잘 정리하고 불필요한 것을 빼는 로직이 없음 — 덧붙이기만 할 뿐

## IronClaw vs OpenClaw: 두 가지 철학

| | IronClaw (Illia) | OpenClaw / OmX (Ultraworkers) |
|---|---|---|
| **철학** | Hardened, Security-first | Recovery-first, 계속 보고 고침 |
| **언어** | Rust | 다양 |
| **강점** | 메모리 누수 없음, 프롬프트 인젝션 방어, 샌드박싱 | 익스플로레이션에 강함, 브레인스토밍하면서 일하기 좋음 |
| **적합한 상황** | 반복 가능하고 예측 가능한 워크플로우, 프로덕션 서빙 | 아직 뭘 해야 할지 탐색 중인 단계 |
| **운영 안정성** | 크론잡 누적, 로그 파일 폭증 같은 운영 이슈를 아키텍처 수준에서 방지 | OpenClaw는 켜놓으면 메모리가 계속 새는 이슈 존재 |

Yeachan의 결론: 탐색과 브레인스토밍은 OpenClaw/OmX로 하고, 반복 가능한 워크플로우가 확정되면 IronClaw 같은 런타임으로 옮겨서 클라우드에 안정적으로 서빙하는 것이 이상적.

## 비용 문제: 무시할 수 없는 현실

Jeff Wang이 짚은 엔터프라이즈 비용 트렌드:

- 기업들이 코딩 도구에 **100만 달러**를 쓸 줄 알았는데, 실제로는 **1,000만 달러**
- 연간 **1억 달러** 이상으로 추세가 향하고 있음
- 개발자 1만 명 기준, 1인당 **월 $5,000** → 연간 **6억 달러**
- 이는 지속 가능하지 않음

대안:
- **오픈소스 모델 + 특화 소형 모델**로의 전환이 불가피
- Opus를 증류(distill)해서 만든 중간 모델을 튜닝해서 쓰는 방식
- 하네스 레이어가 학습하고 최적화하여 올바른 모델로 라우팅하는 것이 비용 절감의 핵심
- Yeachan도 실제로 개발 가재는 GPT-4o를 쓰고 있으며, 클로드를 안 쓰고 있다고 밝힘

## 에이전트 네이티브 시대: Kim Dongkyu의 관점

Kim Dongkyu(K-SKILL)는 에이전트를 코딩 도구가 아닌 **새로운 인터페이스**로 정의했다:

- 웹 → 웹페이지, 스마트폰 → 모바일 앱, **AI → Claw(에이전트 하네스)**
- Claw는 단순한 오픈소스 프로젝트가 아니라 **새로운 OS**
- 그 안의 스킬(앱)이 에이전트가 무엇을 할 수 있는지를 결정

### 안티 에이전트 vs 에이전트 네이티브
- **안티 에이전트 패턴**: 로그인 + PASS 인증 + 15분 세션 타임아웃 + 프론트엔드 수시 변경
- **에이전트 네이티브 패턴**: 공식 API/CLI 제공, JSON으로 데이터 직접 반환, 프론트엔드 안정적

### 한국의 디지털 갈라파고스 = 기회
- 카카오맵, 쿠팡, 네이버, KTX 등 한국 전용 서비스가 에이전트 네이티브가 되면, 전 국민이 한꺼번에 에이전트 시대로 넘어올 수 있음
- 먼저 에이전트 네이티브로 만든 서비스에 훈련 데이터가 쌓이면, 모델 가중치 안에 "쇼핑 = 쿠팡"이 매핑됨 → 선점 효과
- 조직 혁신(AX)에만 집중하지 말고, **서비스 자체를 에이전트 네이티브로 바꾸는 것**이 진짜 경쟁력

### 미래: 디지털에서 피지컬로
- 기상청 스킬로 내일 비가 오는 걸 파악 → 집에 있는 로봇이 우산을 건네줌
- IoT를 넘어 로봇까지 연결되는 에이전트 인터페이스

### Claw는 다음 OS다
- Jeff: "장기적으로 개인 인스턴스가 폰과 컴퓨터에 미리 번들되어 나올 것. 이것이 다음 운영 체제다."

### 엔터프라이즈 vs 개인
- **개인**: 로컬에서 자유롭게, 보안 끄고 써도 됨
- **기업**: 오케스트레이션 패널, 가드레일, 멀티테넌트 격리 필요
- Devin은 VM 안에서 코드 변경 + DB 접근 + MCP 서버 + 웹 브라우저 검증까지 하나의 환경으로 제공

### 비기술 업무의 미래
- 코드는 "실행되는지"로 검증 가능하지만, 인간 판단 업무는 검증이 어려움
- 해법: **작업(task)이 아닌 목표(goal)를 설정**하고 AI가 알아서 방법을 찾게 하는 것
- "콜드 이메일 보내기"가 아니라 "고객 30명 확보"를 목표로 주면, 에이전트가 반복 실험하며 최적화
- 성공 신호(이메일 답장, 가입 등)를 AI가 인식할 수 있도록 설계하는 것이 핵심

### 에이전트 커머스와 TEE
- 에이전트끼리 협상 → 에스크로 → 작업 수행 → 결과 전달의 프레임워크
- Trust Execution Environment에서 실행하면 전체 실행 추적을 암호화 서명으로 검증 가능
- 분쟁 시 "에이전트의 뇌에 GoPro를 달아놓은 것"처럼 과정을 투명하게 검사 가능
- 디지털 긱 워크(Fiverr 등)에서 먼저 적용 → 공급망/물류/조달로 확장 예상

### 피지컬 AI
- Jeff의 thesis: 휴머노이드 로봇은 **다른 인간과 상호작용해야 할 때만** 필요 (소매업, 바리스타 등)
- 공장은 이미 특화 로봇으로 자동화 진행 중 (RaaS: Robot as a Service)
- 헬스케어/노인 돌봄: 생체 정보 감지, 환경 관리 등 비휴머노이드 폼팩터도 중요

## 결론: 경쟁의 전선이 이동했다

이 밋업이 말하는 것은 단순히 "코딩이 자동화됐다"가 아니다.

**모델은 commodity가 되어가고 있고, 진짜 차별화는 실행 레이어에서 일어난다.** 누가 더 좋은 모델을 만드느냐가 아니라, 누가 더 안전하고 안정적이고 비용 효율적으로 에이전트를 실행하고 제어할 수 있느냐가 경쟁의 핵심이 되고 있다.

- **Illia**는 "Attention Is All You Need"를 쓴 사람이 왜 하네스를 만드는가로 이 메시지를 체현하고 있고
- **Jeff**는 엔터프라이즈에서 이것이 어떻게 스케일되는지를 보여주고 있고
- **Yeachan**은 실제로 매일 에이전트를 운영하면서 부딪히는 현실의 교훈을 공유하고 있고
- **Dongkyu**는 이 변화가 코딩을 넘어 일상생활과 산업 전반으로 확장되는 그림을 그리고 있다

하네스 레이어는 단순한 도구가 아니라, AI 시대의 **운영 체제**가 되어가고 있다. 그리고 그 운영 체제의 주도권은 지금 오픈소스 커뮤니티가 쥐고 있다.
